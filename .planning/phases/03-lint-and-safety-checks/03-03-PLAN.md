<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| lint findings -> human/CI output | Human and automated callers rely on stable fields, severity names, and exit codes. |
| test fixtures -> validation claims | Temporary repo fixtures must prove behavior through the same CLI users run. |
| reference docs -> future agent behavior | Documentation can mislead later agents if it still describes lint as deferred or incomplete. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-11 | Repudiation | `render_text_findings` and `render_json_findings` | mitigate | Render the same fixed finding fields in text and JSON so issues are attributable and actionable. |
| T-03-12 | Denial of Service | `lint_exit_code` | mitigate | Centralize exit-code logic so warning-only results do not fail CI or agent workflows. |
| T-03-13 | Tampering | read-only lint checks | mitigate | Add byte-for-byte fixture assertions that warning and error lint runs do not modify wiki files. |
| T-03-14 | Repudiation | TEST-04 and TEST-05 fixtures | mitigate | Add explicit fixtures for missing index entries and secret-looking raw content so roadmap validation is executable. |
| T-03-15 | Information Disclosure | docs and remediation | mitigate | Document redaction/removal remediation for secret-looking content and preserve the raw policy boundary. |
</threat_model>
